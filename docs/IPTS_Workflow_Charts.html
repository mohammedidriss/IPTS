Authentication & Authorization Flow
All API access requires a JWT bearer token. Roles (admin/supervisor/user) control endpoint access via RBAC middleware. Sessions are stateless — no server-side session storage.

Payment Processing Flow — AML + Blockchain Settlement
Every settlement payment passes through a 5-stage pipeline: input validation → 16-feature AML scoring → threshold routing → HITL queue or blockchain atomic settlement → SSE notification.
flowchart TD A([👤 User submits\nSettlement Form]) --> B["POST /api/payments/settle\nFlask — JWT validated"] B --> C[Validate inputs\nbeneficiary, amount,\ncurrency, account] C --> D[Build 16-Feature Vector\nSee AML Pipeline chart] D --> E["AML_Risk_Engine.score()\n5-model ensemble\nXGBoost + RF + IF +\nAutoencoder + Sequence"] E --> F[Compute composite\nrisk_score 0–100\nSHAP values generated] F --> G{Risk Threshold} G -->|score < 30| H[✅ AUTO-APPROVED\nLow risk] G -->|30 ≤ score < 70| I[⚠️ REVIEW QUEUE\nHITL required] G -->|score ≥ 70| J[🚫 AUTO-BLOCKED\nHigh risk] H --> K["Submit to Blockchain\nWeb3.py call\nIPTS_Settlement.settle()"] I --> L[Store in pending_approvals\nSQLite — status: pending] J --> M[Log to transactions\nstatus: blocked\nNo blockchain call] K --> N{Smart Contract\nGanache EVM} N -->|Success| O[Record tx_hash\nDeduct sender balance\nCredit receiver\nLog to audit_trail] N -->|Revert| P[❌ Blockchain error\nRefund sender\nRollback SQLite] L --> Q[Supervisor/Admin\nsees HITL queue] Q --> R{Four-Eyes\nApproval} R -->|1st approval| S[status: partially_approved] S --> T[2nd approval\ndifferent user] T -->|Approved| K T -->|Rejected| U[status: rejected\nNotify sender] O --> V[SSE Push Notification\n/api/stream\nclient receives alert] M --> V U --> V style A fill:#003366,color:#fff style O fill:#1B5E20,color:#fff style J fill:#B71C1C,color:#fff style P fill:#B71C1C,color:#fff style U fill:#E65100,color:#fff

AI/ML Risk Scoring Pipeline
The AML engine builds a 16-dimensional feature vector per transaction, runs it through a 5-model ensemble, and produces a composite risk score with SHAP-driven per-feature explanations. Models are persisted to disk after first training.

HITL Four-Eyes Approval Flow
Transactions scoring 30–70 enter the Human-in-the-Loop queue. Two separate approvers (four-eyes principle) must approve before execution. Neither approver can be the transaction originator.
sequenceDiagram participant U as 👤 User (Sender) participant API as 🔌 Flask API participant DB as 🗄️ SQLite participant S1 as 👔 Supervisor 1 participant S2 as 👔 Supervisor 2 participant BC as ⛓️ Ganache U->>API: POST /api/payments/settle\n{amount: $850K, beneficiary: ...} API->>API: AML Score = 55 (REVIEW) API->>DB: INSERT pending_approvals\nstatus = 'pending' API-->>U: {status: "pending", msg: "Under review"} Note over S1,S2: SSE notification pushed to supervisors S1->>API: GET /api/admin/pending-approvals\n(Bearer: supervisor1_token) API-->>S1: [{id:42, amount:850000, risk:55...}] S1->>API: POST /api/admin/approve/42\n{action: "approve"} API->>DB: UPDATE status = 'partially_approved'\napprover1 = 'supervisor1' API-->>S1: {status: "partially_approved"} Note over S2: Four-eyes: different user required S2->>API: POST /api/admin/approve/42\n{action: "approve"} API->>DB: UPDATE status = 'approved'\napprover2 = 'supervisor2' API->>BC: Web3.py → IPTS_Settlement.settle() BC-->>API: tx_hash = 0xabc... API->>DB: INSERT transactions (settled)\nDEDUCT sender balance\nCREDIT receiver balance API-->>S2: {status: "approved", tx_hash: "0xabc..."} Note over U: SSE push: "Payment settled ✓"

HITL Technologies

  • Flask REST API
  • SQLite pending_approvals
  • JWT RBAC (role check)
  • SSE real-time push
  • Web3.py (on approval)
  • Audit trail logging

Four-Eyes Rules

  • Min. 2 distinct approvers
  • Approver ≠ Sender
  • Supervisor or Admin role
  • Partial → Full approval
  • Rejection notifies sender

DeFi Engine Flows — AMM Swap · Staking · HTLC Escrow
The DeFi Hub implements three protocols: constant-product AMM for FX swaps, a tiered yield farming staking system, and hash time-locked contracts (HTLC) for trustless escrow.

6a · AMM Swap (Constant-Product x·y = k)

flowchart TD A([User enters swap:\nFrom: USD 1000\nTo: EUR]) --> B["GET /api/defi/pools\nFetch pool reserves\nreserve_a, reserve_b"] B --> C["Calculate output:\nout = reserve_b × in\n÷ reserve_a + in × 0.997\nConstant-product AMM"] C --> D["Compute price impact:\nimpact = in ÷ reserve_a × 100"] D --> E{Price impact\n> 5%?} E -->|Yes| F[⚠️ High impact warning\nshown to user] E -->|No| G[Show preview:\nYou get: 921 EUR\nFee: 0.3%\nImpact: 1.2%] F --> G G --> H{User confirms\nswap} H -->|Cancel| I[Abort] H -->|Confirm| J["POST /api/defi/swap\nFlask endpoint"] J --> K[Deduct USD from\nuser balance] J --> L["UPDATE amm_pools\nnew_reserve_a += in\nnew_reserve_b -= out\nSQLite atomic tx"] J --> M["INSERT swap_history\namount_in, amount_out,\nfee, timestamp"] L --> N["Web3.py → LiquidityPool\n.swap() on-chain record"] K & M & N --> O[Credit EUR to\nuser balance] O --> P[✅ Swap complete\nBalance updated in UI] style A fill:#003366,color:#fff style P fill:#1B5E20,color:#fff style I fill:#B71C1C,color:#fff

Network Graph & Fraud Detection Flow
NetworkX builds a transaction graph where nodes are entities and edges are transfers. Betweenness centrality, clustering coefficients, and community detection identify money laundering rings and high-risk hubs.
